# 🚀 Эволюция проектов

Мой путь от простых задач к более сложным архитектурным решениям:

### 1. **[CRUD API Service](https://github.com/SergeyKo17/sprint6-finaltask)** – Базовый уровень
Простой CRUD сервис с 4 основными endpoint'ами. Освоение базовых принципов работы HTTP-сервера в Go, подключение к базе данных и выполнение простых операций.

### 2. **[Advanced CRUDL Service](https://github.com/SergeyKo17/yp-final)** – Промежуточный уровень
Расширенный CRUDL сервис с более сложной бизнес-логикой и простой системой авторизации. Реализована работа с JWT-токенами, добавлены дополнительные операции и валидация. Проект имеет базовое разделение на слои (без middleware) для лучшей организации кода и ReadMe.

### 3. **[Layered API Project](https://github.com/SergeyKo17/tt-ef)** – Текущий уровень (реализован за 1 неделю)
Проект с разделением на слои (presentation, business, data), использованием DTO (структуры для передачи данных между слоями) и более продуманной архитектурой, более подробным ReadMe и Swagger-документацией.

**Анализ последнего проекта выявил конкретные направления для улучшения:**
- Необходимость реализации graceful shutdown для корректного завершения работы
- Понимание, что DTO-структуры нужно проектировать с учетом всех слоев с самого начала
- Требуется доработка части бизнес-логики для полного соответствия требованиям
- Планируется замена стандартного логгера на более функциональное решение
- Хочется добавить builder-паттерн для построения сложных запросов

Эти "болевые точки" определяют мою текущую учебную повестку. Я не просто изучаю технологии абстрактно, а целенаправленно закрываю пробелы, выявленные в реальных проектах. Параллельно углубляю теорию, прохожу мок-интервью с другими развивающимися разработчиками и помогаю учащимся на курсах разбираться в сложных для них темах.
